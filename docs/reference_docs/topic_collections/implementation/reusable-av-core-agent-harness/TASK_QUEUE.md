## Queue rules

1. Allowed statuses are only: `READY`, `IN_PROGRESS`, `PENDING`, `BLOCKED`, `DONE`.
2. Only one task may be `IN_PROGRESS` globally unless a lane rule explicitly allows concurrency.
3. Promote `PENDING` -> `READY` only when all dependencies are satisfied.
4. Selection order is deterministic: highest priority (`P0` -> `P1`) then lowest task ID.
5. External dependency tokens are authoritative from their source queues (YAI/OCG).
6. If a row is `BLOCKED`, write the blocker and move to the next deterministic `READY` row.
7. Every row must run all commands listed in `Verify` before moving to `DONE`.
8. Keep lane ownership strict to minimize merge conflict with `YAI` parity rows.
9. Keep native `vc83` authority precedence non-bypassable for all AV-originated intents.
10. Sync `TASK_QUEUE.md`, `SESSION_PROMPTS.md`, `INDEX.md`, and `MASTER_PLAN.md` after every completed row.
11. Keep task granularity to small, testable slices (target 1-2 hours).

## Dependency token semantics

1. `ID`: dependency must be `DONE` before this row can move `PENDING` -> `READY`.
2. `ID@READY`: dependency must be `READY` or `DONE` before this row can move `PENDING` -> `READY`.
3. `ID@DONE_GATE`: row may become `READY`/`IN_PROGRESS`, but cannot move to `DONE` until dependency is `DONE`.
