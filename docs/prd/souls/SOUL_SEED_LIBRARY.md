## Behavioral-System Contract Alignment (`LOC-007`)

1. A seed `soul` must package the full behavioral stack: `prompt + memory + policy + tools + eval + trust`.
2. `systemPrompt` text is descriptive only; seed-level differentiation must be primarily encoded through stable identity memory, policy/autonomy fields, tool scope, and trust constraints.
3. If a seed cannot declare non-prompt behavioral layers, it is documentation-only and must not be treated as runtime-ready.

---

## Runtime contract notes

Seed rows must align with current runtime boundaries and PRD v1.3 contract updates:

1. Agent records live in `objects` (`type="org_agent"`) with `status: "template"`.
2. Seed ingestion must reuse protected-template upsert + clone-policy mechanics in `convex/onboarding/seedPlatformAgents.ts`; do not create a parallel seed stack.
3. Owner approval stays enabled for mutation pathways by default.
4. One-agent runtime fields (`unifiedPersonality`, `teamAccessMode`, specialist metadata) must be representable in template metadata.
5. Personal-operator template specifics remain owned by `PLO-010`; this doc consumes that contract and does not redefine it.
6. Access modes are runtime contract semantics from `YAI-003`/`YAI-004`, not UI-only labels; `meeting` mode remains advisory unless an explicit approved mutation path is taken.
7. `soulV2.identityAnchors.immutableOrigin` expectations are explicit: core seed templates default to `manual`, while downstream birthing overlays may produce `interview`/`generated` origins via `attachSoulV2Overlay` without mutating immutable identity anchors.
8. `systemPrompt` and `Soul Blend` cannot be the sole behavioral differentiator for any seed promoted toward runtime use.
9. Trust requirements must remain explicit (`requireOwnerApprovalForMutations`, trust-event visibility, and fail-closed mutation posture for missing approvals).
